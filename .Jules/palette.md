## 2024-04-15 - Dynamic ARIA Attributes Syncing
**Learning:** Found that interactive components like language toggles and mobile nav menus had static `aria-label` or missing `aria-expanded` attributes that didn't update when the UI state changed. This creates a disconnect for screen reader users where the visual state (like menu opening or language changing) isn't communicated properly.
**Action:** Always ensure that JavaScript toggles (like `.toggle()`) that change visual state also dynamically update corresponding ARIA attributes (e.g., `aria-expanded`, `aria-label`) so assistive technologies reflect the current state.
## 2024-05-10 - Canvas Animations and prefers-reduced-motion
**Learning:** Found that JavaScript-driven animations (like drawing on a `<canvas>`) are inherently immune to standard CSS `@media (prefers-reduced-motion: reduce)` rules that globally kill CSS transitions and keyframe animations. Users who request reduced motion at the OS level to avoid vestibular discomfort would still experience continuous particle animations.
**Action:** Always verify `window.matchMedia('(prefers-reduced-motion: reduce)').matches` in JavaScript before instantiating heavy/continuous visual animations on the canvas, and hide the canvas element if the user prefers reduced motion.
## 2024-05-01 - Hiding Meaningless Syntax from Screen Readers
**Learning:** Found that purely decorative or visual-thematic text components, such as simulated terminals and code syntax typing effects, are read out completely literally by screen readers. This clutters the auditory experience and confuses visually impaired users trying to navigate actual content.
**Action:** Always add `aria-hidden="true"` to the wrapper elements of simulated terminal UI or code typewriter effects so that assistive technologies skip the meaningless pseudo-code and syntax characters.
## 2024-05-01 - Performant Scrollspy & Mobile Accessibility
**Learning:** Found that scroll event listeners processing navigation states without throttling/RAF can lead to performance degradation (jank) and high CPU usage. Also, mobile navigation menus opened by user actions must provide a way to close them using a keyboard (`Escape` key), to be accessible. Lastly, active items in navigation should use the `aria-current="page"` attribute for screen readers, not just a CSS class.
**Action:** Use `IntersectionObserver` or `requestAnimationFrame` with passive event listeners for scrollspy logic, dynamically apply `aria-current="page"`, and listen for the `Escape` key on open UI components like nav menus to properly dismiss them and restore focus.
